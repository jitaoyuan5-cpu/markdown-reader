# AGENTS.md

## 目的
本仓库使用 Codex，并结合项目级指导与团队共享 skills。
开始任何工作前，先阅读本文件。
当任务与 `.agents/skills` 中的某个 skill 明确匹配时，优先使用对应 skill。

## 团队优先级
在进行修改时，请按以下优先级权衡：
1. 正确性
2. 对现有行为的安全性
3. 改动尽量小且便于 Review
4. 可维护性
5. 交付速度

## 通用工作规则
- 优先选择能完整解决问题的最小改动。
- 除非任务明确要求，不要顺手重构无关代码。
- 除非任务明确允许破坏性变更，否则应保持对外行为和兼容性稳定。
- 在引入新抽象之前，先复用仓库内已有模式。
- 修改前先阅读相关代码路径，不要盲改。
- 当需求不完整时，做出最安全、最合理的假设，并在结果中明确说明。
- 如果改动存在风险，必须明确指出。

## 测试要求
- 任何行为变更原则上都应补充测试，除非确实无法测试或任务范围明确不包含测试。
- 如果没有补测试，说明原因。
- 先跑最小必要验证，再按需要扩大验证范围。
- 没有经过验证，不要声称问题已经完全解决。

## 文档要求
- 如果改动影响公共 API、配置行为、开发流程或发布流程，应同步更新相关文档。
- 如果不需要更新文档，请在结果中简要说明。

## 改动边界
- 除非任务明确要求，不要修改密钥、凭证、生产基础设施或部署配置。
- 不要修改 lockfile、CI 配置、数据库 schema 或生成文件，除非任务明确涉及。
- 不要随意重命名文件、包、公共接口，除非这是完成任务所必需的。

## 代码质量要求
- 保持函数和模块职责清晰、内聚。
- 命名优先清楚直白，不要刻意炫技。
- 避免“为未来可能需要”而提前抽象。
- 错误处理风格应与周边代码保持一致。
- 日志和注释应有实际价值，避免无意义堆砌。

## 默认输出格式
除非用户另有要求，最终输出请包含：
1. 改了什么
2. 为什么这样改
3. 做了哪些验证
4. 风险 / 后续建议

## 本仓库可用 skills
当任务适配时，使用以下 repo-local skills：
- `bugfix`：用于定位并修复缺陷、回归、偶发异常、运行时错误、失败测试
- `code-review`：用于审查 diff、分支、commit、已暂存改动或未提交改动
- `release-check`：用于发布前检查、上线风险评估、识别发布阻塞项
- `feature-implementation`：用于在现有系统中新增明确功能，要求基于现有模式完成最小闭环，并补齐必要测试、文档、配置、权限或发布说明

## Skill 路由提示
当任务类似以下表达时，优先使用 `bugfix`：
- 修这个 bug
- 查一下这个报错
- 找一下回归原因
- 为什么这里失败了
- 帮我定位根因

当任务类似以下表达时，优先使用 `code-review`：
- Review 这个分支
- 看一下这个 diff
- 检查这个 PR
- 审一下这些改动
- 看看这个 commit 有没有风险

当任务类似以下表达时，优先使用 `release-check`：
- 这次能不能发版
- 帮我检查发版准备
- 看看还有什么阻塞上线
- 做一下发版前检查
- 验证这个 release 分支是否可发布

当任务类似以下表达时，优先使用 `feature-implementation`：
- 新增一个功能
- 在现有页面上加一个能力
- 给这个接口扩展一个新行为
- 增加一个后台操作流程
- 在原有应用上补一个业务功能

## 使用 skill 时的偏好行为
- 如果某个 repo-local skill 明确匹配当前任务，应优先使用。
- 使用 skill 时，遵循该 skill 规定的输出结构。
- 输出应尽量简洁、基于证据、便于执行。
