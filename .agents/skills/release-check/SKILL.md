# release-check skill

当任务目标是判断是否具备发布条件，并提前识别阻塞项和上线风险时，使用此 skill。

## 目标
判断当前改动是否可以发布，识别阻塞项，并在上线前暴露兼容性和运维风险。

## 工作流程

### 1. 明确发布范围
先识别：
- 这次要发布什么
- 涉及哪些组件
- 这是热修复、常规发布、migration 还是纯配置变更
- 是否有对用户可见或对运维可见的行为变化

### 2. 检查发布阻塞项
重点审查以下维度：

#### 正确性与完整性
- 预期改动是否已经真正实现？
- 是否存在明显未完成实现、TODO、占位逻辑或半成品 migration？

#### 验证情况
- 相关测试是否已更新或执行？
- 关键路径是否缺少必要验证？

#### 兼容性
- 是否存在 API、schema、配置或行为层面的破坏性变化？
- 是否可能影响依赖方、调用方或其他服务？

#### 数据 / migration 风险
- 是否涉及 schema 或数据迁移？
- 是否需要 backfill？
- 回滚是否困难？

#### 配置 / 环境风险
- 是否新增环境变量、密钥、feature flag 或基础设施前提？
- 默认值是否安全？

#### 运维准备度
- 可观测性是否足够？
- 出问题时是否容易诊断？
- 是否有重试、超时、队列、定时任务或并发方面的风险？

#### 文档与发布说明
- 是否需要更新 runbook、迁移说明、运维指引或发布说明？
- 是否建议灰度、开关控制或监控窗口？

### 3. 给出发布结论
结论只能是以下三种之一：
- Ready
- Ready with caveats
- Not ready

应基于证据做出偏保守判断。

## 审查原则
- 重点关注“能不能安全上线”，而不是泛化代码风格问题。
- 优先识别真正阻塞发布的风险。
- 如果验证证据不足，要明确指出不确定性。

## 最终输出要求
除非用户另有要求，按以下结构输出：

### Release status
- `Ready`、`Ready with caveats` 或 `Not ready`

### Blocking issues
- 项目符号列表；如果没有，写 `None`

### Caveats / watch items
- 项目符号列表；如果没有，写 `None`

### Recommended checks before release
- 发布前最值得补做的检查项列表

### Rollout notes
- 说明 migration、监控、feature flag、回滚或沟通层面的注意事项
