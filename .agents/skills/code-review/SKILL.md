# code-review skill

当任务目标是系统性审查一组改动并给出可执行问题清单时，使用此 skill。

## 目标
识别一组改动中最值得关注的问题，优先关注正确性、回归风险和验证缺口。

## 审查原则
以证据为基础，优先输出高价值问题，不要堆砌低价值噪音。
除非会显著影响可读性或维护性，否则不要沉迷于琐碎格式问题。

## 审查流程

### 1. 理解改动意图
先判断这组改动想解决什么问题，例如：
- 修 bug
- 新功能
- 重构
- migration
- 仅测试改动
- 配置 / 发布相关改动

如果意图不明确，应从 commit 信息、变更文件、测试和代码结构中推断。

### 2. 审查范围
检查：
- 改动涉及了哪些模块
- 是否意外跨越了不该跨的边界
- diff 是否明显大于必要范围
- 是否混入了无关文件修改

### 3. 按风险维度审查
按以下顺序评估：

#### 正确性
- 逻辑是否真的实现了它看起来要做的事？
- 边界情况是否处理到位？
- 失败路径是否考虑过？
- 关键假设是否被验证？

#### 兼容性
- 是否破坏了公共 API、schema、约定或既有行为？
- 调用方 / 使用方是否同步更新？

#### 回归风险
- 是否改动了共享路径或高复用逻辑？
- 是否可能无意影响现有流程？
- 是否存在隐藏行为变化？

#### 测试与验证
- 行为变更是否补了测试或更新了测试？
- 测试是否真正覆盖高风险路径？
- 是否缺少反例、异常路径或边界值覆盖？

#### 可维护性
- 设计是否容易理解？
- 是否符合仓库现有模式？
- 命名是否清晰？
- 复杂度是否合理？

#### 安全与可靠性
- 是否有输入校验缺口？
- 是否有认证 / 授权错误？
- 是否有敏感信息泄露风险？
- 是否存在不安全的 shell / SQL / 文件 / 网络操作？

#### 性能与运维
- 是否存在明显热点路径问题？
- 是否有重试、超时、并发风险？
- 是否削弱了日志、监控或可观测性？

### 4. 归纳并排序问题
只报告真正值得作者关注的问题。
优先少而强的问题，不要列很多弱问题。

## 严重级别
使用以下分级：

### Blocker
阻塞合并或阻塞发布的问题。通常意味着逻辑错误、风险过高或实现明显不完整。

### Major
重要问题，通常应在合并前修复，但未必是灾难性问题。

### Minor
值得改进的问题或防护缺口，通常不一定阻塞合并。

## 最终输出要求
除非用户另有要求，按以下结构输出：

### 总结
- 用 1 到 3 句话概括整体质量和风险。

### Blockers
- 项目符号列表；如果没有，写 `None`。

### Major issues
- 项目符号列表；如果没有，写 `None`。

### Minor issues
- 项目符号列表；如果没有，写 `None`。

### Suggested follow-up
- 简要列出还能降低不确定性的测试、检查或需补充确认的点。
