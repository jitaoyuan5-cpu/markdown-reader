# feature-implementation skill

当任务目标是在现有系统上新增一个功能，并以团队既有模式完成落地时，使用此 skill。

## 目标
在不破坏现有系统稳定性的前提下，基于仓库已有结构实现一个清晰、完整、可验证的最小功能闭环，并明确说明影响范围与后续风险。

## 适用场景
适用于以下类型任务：
- 在现有页面中新增一个功能入口、筛选项、表单能力或交互流程
- 在现有后端服务中新增一个接口、业务分支或处理逻辑
- 在现有管理后台中新增菜单、权限点、操作能力或状态流转
- 在现有模块中扩展字段流转、校验逻辑、展示逻辑或业务规则
- 在已有功能基础上补一个明确范围的新能力

不适用于：
- 单纯修 bug 或排查异常
- 只做代码审查
- 只做发布前检查
- 大范围架构改造或完全从零搭建新系统

## 工作原则
- 优先复用现有模式，不轻易引入新范式。
- 优先做最小完整闭环，不做任务之外的扩展。
- 明确区分“本次范围内”与“本次不做”。
- 尽量保持接口、模块边界和代码风格一致。
- 新增能力时，同步考虑测试、文档、配置、权限、埋点、迁移和上线影响。

## 工作流程

### 1. 理解需求与边界
在动手前，先明确：
- 本次新增的功能是什么
- 用户或调用方能看到什么变化
- 哪些行为属于本次范围
- 哪些需求不在本次范围内
- 是否存在兼容性约束或上线约束

如果需求描述不完整，应基于现有代码、模块职责、已有交互和仓库惯例，做出最安全、最合理的假设，并在最终结果中说明。

### 2. 先研究现有实现
不要直接发明新写法。先检查：
- 仓库里是否已有类似功能
- 当前模块的目录结构、分层方式、命名方式
- 相关接口、DTO、类型定义、状态管理、权限控制、表单处理、埋点方式
- 现有测试如何组织
- 是否已有可复用组件、工具函数、校验逻辑或服务层

优先沿用仓库已有模式，再考虑新增抽象。

### 3. 设计最小实现路径
在正式修改前，先收敛出最小闭环：
- 入口在哪里
- 核心逻辑改哪里
- 数据如何流转
- 哪些调用方会受影响
- 需要新增哪些文件或测试
- 是否需要更新文档、配置、权限、埋点、migration 或发布说明

避免“顺手再做一点”。除非明确必要，不扩展任务范围。

### 4. 实现功能
实现时应重点关注：
- 是否与现有模块职责一致
- 是否保持对外行为兼容
- 是否存在边界条件遗漏
- 是否需要输入校验、错误处理、空值处理、状态同步
- 是否会影响共享路径或公共组件
- 是否需要 feature flag、灰度控制或兼容分支逻辑

### 5. 补齐配套项
根据功能类型，检查是否需要同步补齐以下内容：

#### 测试
- 单测
- 集成测试
- 关键路径验证
- 反例或边界情况覆盖

#### 文档
- 公共 API 文档
- 开发说明
- 使用说明
- 配置说明
- 发布说明

#### 配置
- 环境变量
- 默认值
- 开关项
- 兼容旧配置逻辑

#### 权限
- 菜单权限
- 操作权限
- 角色判断
- 服务端授权校验

#### 埋点 / 日志 / 可观测性
- 是否需要埋点
- 是否需要补充关键日志
- 是否便于后续定位问题
- 是否会影响监控指标

#### 数据与迁移
- 是否涉及 schema 变更
- 是否需要 migration
- 是否需要 backfill
- 是否存在回滚难度

### 6. 验证
优先执行与新增功能最相关、最有价值的验证：
- 新增或更新的测试
- 与功能直接相关的关键流程验证
- 必要时跑 lint / typecheck / build
- 如果功能影响接口或页面，验证关键成功路径和主要失败路径

如果无法完整验证，要明确说明哪些内容未验证。

### 7. 检查对现有系统的影响
在完成实现后，快速检查：
- 是否引入不必要的行为变化
- 是否破坏现有兼容性
- 是否影响共享模块
- 是否有明显性能、并发、权限或发布风险

## 修改规则
- 优先最小闭环，不做无关重构。
- 能沿用现有实现方式时，不要新开体系。
- 除非任务明确要求，不要改公共接口契约。
- 不要把无关优化、代码清理、命名调整混入本次功能改动。
- 对新增能力的限制条件、默认行为和边界处理要明确。

## 最终输出要求
除非用户另有要求，按以下结构输出：

### 功能目标
- 用 1 到 3 句话说明本次新增了什么能力，以及适用范围。

### 实现方案
- 说明核心改动点、涉及模块、数据或调用链路如何变化。
- 如有关键设计取舍，简要说明原因。

### 配套补充
- 说明是否补了测试、文档、配置、权限、埋点、migration 或发布说明。
- 如果某项不需要补，简要说明。

### 验证情况
- 列出实际做过的检查。
- 如果没有完全验证，说明哪些未验证。

### 风险 / 后续建议
- 说明可能的边界问题、上线注意事项、灰度建议或值得继续补强的地方。
