# bugfix skill

当任务目标是定位问题原因并以最小、安全的方式修复时，使用此 skill。

## 目标
给出正确修复，并清晰说明根因与验证结果。

## 工作流程

### 1. 明确问题
在修改代码前，先识别：
- 失败的是什么
- 预期行为是什么
- 实际观察到的行为是什么
- 可能影响的范围是什么

如果问题描述比较模糊，应结合代码、测试、日志、报错堆栈等信息，推断出最安全、最合理的理解。

### 2. 复现或缩小范围
优先尝试直接复现问题。
如果无法完整复现，也要利用现有证据尽量缩小问题范围，例如：
- 失败测试
- 报错堆栈
- 日志
- 最近改动
- 代码路径检查
- 类型或接口不匹配
- 边界条件

在没有缩小失败路径之前，不要直接开始改代码。

### 3. 找到根因
要找“真正原因”，而不是只改表面症状。
重点检查：
- 错误前提假设
- null / 空值 / 缺失输入处理
- 异步时序和竞争条件
- 过期状态 / 缓存交互
- 边界值 / off-by-one 问题
- schema / 接口漂移
- feature flag / 配置不一致
- 最近重构引入的回归

### 4. 设计最小且安全的修复方案
优先使用能解决根因的最小改动。
除非为了保证修复安全，否则不要顺手做额外重构。

### 5. 验证
先做最小但有效的验证：
- 针对性单测
- 原本失败的测试
- 与该问题直接相关的集成路径
- 如果相关，再跑 lint / typecheck

如果验证无法完成，要明确说明还有哪些内容未被验证。

### 6. 检查邻近回归风险
快速检查相邻逻辑，确认修复不会明显引入新的问题。

## 修改规则
- 除修复缺陷本身外，不要改变现有公共行为。
- 保持 diff 小而可审查。
- 能补测试时尽量补测试。
- 遵循仓库现有代码风格与模式。
- 不要把无关清理混进同一个改动里。

## 最终输出要求
除非用户另有要求，按以下结构输出：

### 根因
- 用 1 到 2 句话说明真正原因。

### 修复方案
- 说明改了什么，以及为什么这样能解决根因。

### 验证情况
- 列出实际做过的检查。
- 如果没有完全验证，说明哪些没有跑。

### 风险 / 后续建议
- 说明剩余边界情况、上线注意点或值得补充的检查。
